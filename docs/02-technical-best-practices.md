# Technical Best Practices

## ESPHome Configuration

### File Organization

Use ESPHome's packages system to keep configs DRY. The structure:

```
esphome/
├── secrets.yaml                    # WiFi, API keys, Signal numbers
├── .base-common.yaml               # WiFi, API, OTA, logger (all devices)
├── .base-button.yaml               # Button-specific: deep sleep, LED, press event
├── .base-display.yaml              # Display-specific: SPI, fonts, lambda
├── button-ryker-laundry.yaml       # Per-device config
├── button-ryker-teeth.yaml
├── button-logan-laundry.yaml
├── ...
└── display-front-door.yaml
```

Dot-prefix on base files (`.base-*`) keeps them grouped at top of directory listings and signals "don't flash this directly."

### Base Common Config (.base-common.yaml)

Every device inherits this. Contains:

```yaml
wifi:
  ssid: !secret wifi_ssid
  password: !secret wifi_password
  ap:
    ssid: "${device_name} Fallback"
    password: !secret fallback_password

api:
  encryption:
    key: !secret api_encryption_key

ota:
  - platform: esphome
    password: !secret ota_password

logger:
  level: INFO
```

### Per-Device Config Pattern

Each device YAML is minimal — just substitutions and package includes:

```yaml
substitutions:
  device_name: button-ryker-laundry
  friendly_name: "Ryker Laundry Button"
  task_owner: ryker
  task_name: laundry

packages:
  common: !include .base-common.yaml
  button: !include .base-button.yaml
```

This means adding a new button = copying a 10-line file and changing 4 substitution values.

### Substitutions as the Interface

Substitutions are the contract between base configs and per-device configs. Define these clearly:

| Substitution | Used In | Example |
|---|---|---|
| `${device_name}` | Common (hostname, AP name) | `button-ryker-laundry` |
| `${friendly_name}` | Common (HA display name) | `Ryker Laundry Button` |
| `${task_owner}` | Button base (event data) | `ryker` |
| `${task_name}` | Button base (event data) | `laundry` |

### Naming Conventions (ESPHome)

**Device names:** `{type}-{owner}-{task}` using hyphens (RFC1912 compliant)
- `button-ryker-laundry`
- `button-logan-teeth`
- `display-front-door`

**Entity IDs (auto-generated by HA):** `{domain}.{device_name}_{function}`
- `binary_sensor.button_ryker_laundry_press`
- `sensor.button_ryker_laundry_battery`

### Deep Sleep for Buttons

Buttons spend 99.9% of their time asleep. Config pattern:

```yaml
deep_sleep:
  id: deep_sleep_control
  run_duration: 10s        # Max awake time per press
  sleep_duration: 0s       # Sleep indefinitely (wake on pin)
  wakeup_pin:
    number: GPIO${button_pin}
    mode: INPUT_PULLUP
    inverted: true
```

Key rules:
- **10 second max awake time** — connect WiFi, fire event, get confirmation, sleep
- **No periodic wake** — only wake on physical button press
- **Fallback** — if WiFi fails within 10s, sleep anyway (don't drain battery)

### LED Feedback Pattern

```yaml
light:
  - platform: status_led
    id: status_led
    pin: GPIO${led_pin}

# In button press handler:
on_press:
  then:
    - homeassistant.event:
        event: esphome.task_button_press
        data:
          owner: ${task_owner}
          task: ${task_name}
    - wait_until:
        condition:
          api.connected
        timeout: 8s
    - light.turn_on:
        id: status_led
        effect: pulse    # Confirmation flash
    - delay: 1s
    - deep_sleep.enter: deep_sleep_control
```

### E-Ink Display Best Practices

**Pin configuration for Waveshare 7.5" V2 with ESP32 driver board:**

```yaml
spi:
  clk_pin: GPIO13
  mosi_pin: GPIO14

display:
  - platform: waveshare_epaper
    id: eink_display
    model: 7.50inV2      # Confirm exact model on purchase
    cs_pin: GPIO15
    dc_pin: GPIO27
    busy_pin:
      number: GPIO25
      inverted: true      # REQUIRED for 7.50in V2 — prevents display damage
    reset_pin: GPIO26
    update_interval: never  # Only update on HA trigger, not on timer
    lambda: |-
      // Display rendering code here
```

**Critical rules:**
- `busy_pin: inverted: true` is MANDATORY for 7.50in V2 models. Omitting this can damage the display.
- Set `update_interval: never` — we push updates from HA, not on a timer
- Use `full_update_every: 30` for partial refresh support if available on your model
- Keep fonts to 2-3 sizes max to conserve flash memory

### Secrets Management

Single `secrets.yaml` in the ESPHome config directory:

```yaml
wifi_ssid: "YourNetworkName"
wifi_password: "YourWiFiPassword"
fallback_password: "FallbackAP123"
api_encryption_key: "base64-encoded-32-byte-key"
ota_password: "OtaUpdatePassword"
```

**Rules:**
- Never commit secrets.yaml to version control
- Generate API encryption key with: `python3 -c "import secrets, base64; print(base64.b64encode(secrets.token_bytes(32)).decode())"`
- All devices share the same API key (simplifies HA integration)
- OTA password is separate from API key

## Home Assistant Configuration

### Entity Naming Conventions

**Pattern:** `{domain}.task_{owner}_{taskname}`

| Entity | Example |
|---|---|
| Completion state | `input_boolean.task_ryker_laundry` |
| Completion time | `input_datetime.task_ryker_laundry_time` |
| Template sensor | `sensor.task_ryker_remaining` |

**Labels:** Apply HA labels for filtering:
- `task-tracker` — all task tracker entities
- `task-ryker` — Ryker-specific
- `task-logan` — Logan-specific
- `task-daily` — Daily tasks
- `task-wednesday` — Wednesday-only tasks

### Automation Naming

**Pattern:** `task_{trigger_type}_{scope}_{action}`

| Automation | Purpose |
|---|---|
| `task_button_ryker_laundry` | Handle Ryker's laundry button press |
| `task_button_logan_teeth` | Handle Logan's teeth button press |
| `task_reset_daily` | 3 AM daily reset of all tasks |
| `task_display_refresh` | Refresh e-ink on any state change |
| `task_notify_completion` | Send Signal message on task completion |
| `task_notify_allcomplete_ryker` | Notify when Ryker finishes all tasks |

### Automation Structure

Every button automation follows this pattern:

```yaml
alias: "task_button_ryker_laundry"
description: "Handle Ryker laundry button press"
trigger:
  - platform: event
    event_type: esphome.task_button_press
    event_data:
      owner: ryker
      task: laundry
condition:
  # Don't double-complete
  - condition: state
    entity_id: input_boolean.task_ryker_laundry
    state: "off"
  # Check schedule (daily tasks skip this)
  # Wednesday-only tasks add: condition.time weekday: wed
action:
  - service: input_boolean.turn_on
    entity_id: input_boolean.task_ryker_laundry
  - service: input_datetime.set_datetime
    entity_id: input_datetime.task_ryker_laundry_time
    data:
      datetime: "{{ now().strftime('%Y-%m-%d %H:%M:%S') }}"
  - service: notify.signal
    data:
      message: "✅ Ryker put laundry in the chute!"
  # Display refresh happens via separate automation triggered by state change
```

### Template Sensors

Remaining-tasks counters:

```yaml
template:
  - sensor:
      - name: "task_ryker_remaining"
        state: >
          {% set tasks = ['laundry', 'teeth', 'plates', 'pills'] %}
          {% set wed_tasks = ['flute'] if now().weekday() == 2 else [] %}
          {% set all_tasks = tasks + wed_tasks %}
          {% set done = all_tasks | select('is_state', 'on') | list | count %}
          {{ all_tasks | count - done }}
```

### Input Entity Definitions

Group all task tracker inputs in a dedicated YAML file:

```yaml
# configuration.yaml
input_boolean: !include input_booleans.yaml
input_datetime: !include input_datetimes.yaml
```

This keeps the main config clean and makes the task registry inspectable in one place.

### HA Areas and Device Assignment

Create areas matching physical locations:
- **Hall** — laundry buttons
- **Bathroom** — teeth buttons
- **Kitchen** — plates buttons
- **Bedroom** — pills button
- **Music Room** (or wherever instruments live) — flute, trumpet buttons
- **Front Door** — display

Assign each ESPHome device to its area on first integration. This enables area-based automations later.

## Signal Integration

### Docker Setup

```yaml
# docker-compose.yml (add to existing HA stack or standalone)
services:
  signal-cli-rest-api:
    image: bbernhard/signal-cli-rest-api:latest
    container_name: signal-api
    restart: unless-stopped
    environment:
      - MODE=json-rpc
    ports:
      - "8080:8080"
    volumes:
      - ./signal-cli-config:/home/.local/share/signal-cli
```

### HA Configuration

```yaml
# configuration.yaml
notify:
  - name: signal
    platform: signal_messenger
    url: "http://signal-api:8080"  # or http://localhost:8080 if same host
    number: !secret signal_sender_number
    recipients:
      - !secret signal_household_group
```

### Message Conventions

Keep messages short and scannable. Pattern:

- **Task complete:** `✅ {Child} {past-tense action}!` → `✅ Ryker put laundry in the chute!`
- **All tasks done:** `🎉 {Child} finished all {morning/evening} tasks!`
- **System alert:** `⚠️ Task tracker: {issue description}`

Avoid:
- Timestamps in messages (Signal adds its own)
- Overly verbose messages (this is a notification, not a report)
- Messages for already-completed duplicate presses (handled by condition check)

### Signal Group Setup

Create a dedicated Signal group for task notifications. This keeps household chatter separate from tracker noise. Register the group ID in secrets.yaml.

## Error Handling

### Button fails to connect to WiFi
ESPHome's 10-second run_duration ensures the button goes back to sleep. Battery is preserved. Task simply doesn't register — kid can press again.

### Display stops updating
E-ink retains last image. Stale data is visible and obvious (yesterday's date). HA logs will show connection failure. Physical power cycle of display ESP32 resolves most issues.

### Signal API goes down
Notifications fail silently from HA's perspective. Tasks still register in HA state and on the display. Signal is a nice-to-have, not a critical path.

### Duplicate button press
Condition check (`state: "off"`) in HA automation prevents double-counting. Button LED could flash red/orange to indicate "already done."
